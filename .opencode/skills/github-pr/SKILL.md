# GitHub PR 제출 절차

이 skill은 코드 변경이 완료되고 모든 검증 게이트를 통과한 후 PR을 생성할 때 적용한다.

---

## 사전 체크 (모든 항목 PASS 필요)

- [ ] Gate 1 (타입체크/린트/빌드) 통과: `pnpm typecheck && pnpm lint && pnpm build`
- [ ] Gate 2 (유닛 테스트) 통과: `pnpm test`
- [ ] Gate 3 (해당 시):
  - 프론트 변경 → Playwright E2E 통과 (`pnpm test:e2e`)
  - Socket.IO 변경 → 동시성/reconnect 테스트 통과
  - DB 변경 → 마이그레이션 dry-run 성공
- [ ] `git diff --stat`에 spec과 무관한 파일 없음
- [ ] 새 secret/credential commit하지 않음 (`.env`, `*.key`, `*.pem` 등 검사)
- [ ] 새 의존성 추가했다면 spec에 명시된 것

하나라도 fail이면 PR 생성 금지, 루프로 복귀.

---

## 브랜치 전략

다음 패턴 유지:

```text
feature/<task-id>-<short-desc>     # 새 기능
fix/<task-id>-<short-desc>         # 버그 수정
refactor/<task-id>-<short-desc>    # 리팩토링
chore/<task-id>-<short-desc>       # 설정/문서 등
```

예시:
- `feature/CHAT-12-typing-indicator`
- `fix/CHAT-15-reconnect-room-loss`
- `refactor/CHAT-20-message-repository`

## Commit 규칙

### 컨벤션
```text
feat: 기능 추가
fix: 버그 수정
refactor: 리팩토링 (동작 변경 없음)
chore: 설정 변경
docs: 문서 수정
test: 테스트 추가/수정
```

scope가 명확하면 추가: `feat(socket): typing indicator 이벤트 추가`

### 규칙
- **반드시 한국어로 작성**
- 작은 단위 commit 유지
- 한 commit에는 하나의 논리적 변경만 포함
- 너무 짧거나 의미 없는 메시지 금지 (예: "수정", "버그 픽스", "wip")

### 예시
```text
✅ feat: 채팅 메시지 전송 기능 추가
✅ fix: socket reconnect 시 이전 room 복원 안 되던 문제 수정
✅ refactor: room service의 멤버 관리 로직을 RoomManager로 분리
✅ test: room isolation 통합 테스트 추가

❌ fix: 버그 수정 (무엇을?)
❌ update (커밋 컨벤션 위배)
❌ wip
```

### 본문이 필요한 경우
복잡한 변경은 body에 이유와 영향 설명:
```text
fix: socket reconnect 시 이전 room 복원 안 되던 문제 수정

기존 코드는 reconnect 후 클라이언트가 명시적으로 room:rejoin을
호출해야 room이 복원되었음. 이로 인해 네트워크가 잠시 끊긴 사용자가
계속 broadcast를 받지 못하는 문제 발생.

이제 클라이언트는 localStorage에 마지막 room ID를 저장하고,
connect 이벤트 후 자동으로 rejoin을 시도한다.

Resolves: CHAT-15
```

## PR 생성 (GitHub MCP 사용)

도구: `github.create_pull_request`

매개변수:
- `base`: spec에 명시된 base branch (보통 `develop`, 명시 없으면 `main`)
- `head`: 현재 작업 브랜치
- `title`: 한국어로 명확한 요약
- `body`: 아래 템플릿 사용
- `draft`: false (검증 완료 상태이므로)

### PR 제목 규칙
- **반드시 한국어로 작성**
- 변경 내용을 명확하게 요약
- type prefix는 선택 (`[feat]`, `[fix]` 등 - 프로젝트 컨벤션 따름)

예시:
```text
✅ 채팅방 join 로직 개선
✅ Socket reconnect 안정성 개선
✅ 메시지 저장 구조 리팩토링
✅ [fix] 동시 접속 시 멤버 카운트 어긋나는 문제

❌ Improve chat room join
❌ Fix bug
❌ wip
```

### PR 본문 템플릿 (반드시 한국어)

```markdown
## 변경 내용
- 무엇을 변경했는지 bullet point로 명확히 기술
- 추가된 파일, 수정된 파일, 삭제된 파일을 구분해서 설명
- 새 의존성이 있다면 명시

## 이유
- 왜 이 변경이 필요했는지 설명
- 관련 task spec 또는 이슈 참조
- 다른 접근을 검토했다면 왜 이 방법을 택했는지

## 테스트
- 어떤 테스트를 추가/수정했는지
- 어떤 시나리오를 검증했는지
- 자동 테스트로 커버 안 되는 부분이 있다면 수동 검증 절차

## 기타
- 리뷰어가 특히 봐야 할 부분
- 트레이드오프나 알려진 제약
- 후속 작업 (이번 PR 범위 밖이지만 필요한 것)
```

### 예시 PR

```markdown
## 변경 내용
- `backend/src/socket/RoomManager.ts` 신규 추가 (room 멤버 관리 로직 분리)
- `backend/src/socket/chatHandler.ts`에서 RoomManager를 사용하도록 수정
- `backend/src/socket/__tests__/RoomManager.test.ts` 신규 추가
- 동시 join 시 race condition 처리 (`Promise.all` 대신 순차 처리)

## 이유
기존에는 chatHandler 내부에 멤버 관리 로직이 인라인으로 있어
같은 로직이 join/leave/disconnect 핸들러에 중복되어 있었음.
또한 동시에 같은 room에 N명이 join할 때 멤버 카운트가 어긋나는
race condition이 발생함 (CHAT-15 이슈).

RoomManager로 분리하면서 mutex 패턴으로 동시성 문제도 해결.

## 테스트
- 유닛: RoomManager의 join/leave/get 동작 검증 (12개 케이스)
- 통합: 10개 클라이언트가 동시에 같은 room에 join → 멤버 카운트 정확
- 회귀: 기존 chat.spec.ts 모두 통과

## 기타
- RoomManager는 향후 Redis adapter 도입 시 인터페이스만 유지하고
  구현체만 교체하면 됨
- typing indicator 정리 로직은 이번 PR 범위 밖, 별도 task로 분리
```

## PR 생성 후

- PR URL을 사용자에게 보고
- `.agent/state.md`에 task 완료 표시
- 작업 브랜치는 그대로 두고 main으로 checkout
- **auto-merge 활성화 금지** - 사람 리뷰 대기

---

## 금지 사항

- main/master 브랜치로의 직접 commit/push
- `--force` push (rebase가 필요하면 사용자에게 위임)
- 다른 사람의 PR에 commit 추가
- 이미 머지된 PR의 리오픈
- GitHub Actions 워크플로 파일(`.github/workflows/*`) 수정 (spec에 명시된 경우 제외)
- 영어로 commit 메시지/PR 작성 (한국어 강제)

---

## 자주 발생하는 함정

- **타입 prefix 누락**: `채팅 기능 추가` ❌ → `feat: 채팅 기능 추가` ✅
- **여러 변경을 한 commit에**: 기능 추가 + 리팩토링 + 버그 픽스가 한 commit에 → 분리
- **거대한 PR**: 1000줄 넘으면 리뷰 어려움. spec을 더 작게 쪼개야 했다는 신호.
- **테스트 누락**: 새 기능인데 테스트 추가 없음 → 리뷰 거절 사유
- **PR description 부실**: "기능 추가했음" 같은 한 줄 → 템플릿 모든 섹션 채워야

---

## 체크리스트 (PR 생성 직전 self-review)

- [ ] 브랜치명이 컨벤션 따르는가 (`feature/*`, `fix/*` 등)
- [ ] 모든 commit이 한국어이고 type prefix가 있는가
- [ ] 한 commit에 하나의 논리적 변경만 있는가
- [ ] PR 제목이 한국어이고 변경 요약이 명확한가
- [ ] PR 본문에 4개 섹션(변경 내용 / 이유 / 테스트 / 기타) 모두 채웠는가
- [ ] base branch가 spec에 명시된 것과 일치하는가
- [ ] draft가 아닌 ready for review 상태인가
- [ ] auto-merge가 비활성 상태인가
